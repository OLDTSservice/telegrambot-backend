Telegram 機器人後台管理系統 — 功能說明文件

涵蓋機器人判斷機制、條件限制、以及網頁後台每個頁面的按鍵功能。內容依據目前程式碼實際邏輯整理。

技術架構：FastAPI（後端）＋ React / Ant Design（前端）＋ SQLite（資料庫）＋ python-telegram-bot（輪詢接收訊息）＋ Claude Haiku（知識庫問答 AI）

系統架構總覽

本系統是一套可同時管理「多個 Telegram 機器人」的後台，每個機器人各自獨立運作、各自綁定關鍵字規則與知識庫。除了 Telegram，系統也另外支援 Microsoft Teams 機器人模組（架構相同、頁面獨立），本文件僅聚焦於 Telegram 機器人 相關功能。

運作方式

  • 每個機器人啟用後，後端會用 python-telegram-bot 以獨立執行緒對 Telegram 伺服器進行 長輪詢（polling），收到訊息就丟進判斷流程。
  • 所有規則、知識庫內容、對話記錄都存在同一個 SQLite 資料庫，公司與家裡的本地資料庫各自獨立，只有部署在 Render 上的雲端版本才是實際對外運作的機器人。
  • 知識庫問答由 Anthropic Claude（claude-haiku-4-5）負責，机器人不會憑空回答，只會根據知識庫內容作答。

帳號與權限

後台登入採帳號密碼＋JWT Token，Token 有效期 24 小時。共分三種角色：

角色可檢視可新增／編輯／切換開關可刪除可管理帳號（人員）
superadmin 超級管理員✅✅✅✅
editor 編輯員✅✅✅❌
viewer 檢視者✅❌（按鈕會反灰）❌❌
預設帳號 系統初次啟動會自動建立一組 superadmin 帳號 admin / admin123，建議上線後盡快至「帳號管理」頁面更改密碼或新增正式帳號。

機器人管理頁面

側邊選單：Telegram 機器人 → 機器人管理

此頁分為三個分頁（Tab）：機器人管理、群組管理、AI機器人救援。

機器人管理 Tab

管理所有已串接的 Telegram Bot（一個 Bot Token 對應一個機器人身份）。

按鍵 / 欄位功能說明
新增機器人輸入名稱與從 @BotFather 取得的 Token，建立後立即啟動輪詢（polling）開始接收訊息。Token 不可與現有機器人重複。
啟用開關（Switch）關閉時會呼叫 stop_bot() 停止該機器人的 Telegram 輪詢執行緒；重新開啟則呼叫 start_bot() 重新連線。停用中的機器人完全不會收到／處理任何訊息。
編輯（鉛筆圖示）修改機器人名稱或 Token。
刪除（垃圾桶圖示）停止該機器人並自資料庫移除；連動的規則、知識庫文件、對話紀錄需另外處理（不會自動一併刪除）。有二次確認彈窗。
Token 欄位顯示畫面上僅顯示前 20 碼並以 … 省略，避免明碼外洩。
機器人「運行中」標籤 列表上的綠色「運行中」標籤只代表資料庫 is_enabled=true，若要確認實際輪詢是否存活，需透過 GET /api/bots/{bot_id}/status 或直接觀察是否有訊息進來；若 Token 失效或網路異常，輪詢執行緒可能啟動失敗但畫面仍顯示已啟用。此時可用「重新啟動」（API 提供、目前前端未顯示對應按鈕）排解。

群組管理 Tab

列出指定機器人「曾經收到訊息」的所有 Telegram 群組／聊天室，並可個別控制每個群組的 AI 問答及白名單廠商驗證設定。

按鍵 / 欄位功能說明
選擇機器人（下拉選單）先選機器人才會載入其群組清單。
搜尋群組名稱依群組名稱關鍵字篩選。
AI 問答 開關此開關直接對應 TelegramGroupSetting.ai_enabled（預設為開啟）。關閉後，該群組的訊息完全不會查詢知識庫，機器人只剩「關鍵字規則」仍會作用；此設定也是觸發「AI 救援」的先決條件之一，詳見 訊息流程 — 群組 AI 開關。
群組類型標籤私聊 / 群組 / 超級群組 / 頻道，取自 Telegram 原生 chat.type。
白名單廠商驗證 欄（Tag + 🛡 按鈕） 顯示該群組廠商驗證功能的狀態（「已啟用」藍色標籤 / 「未啟用」灰色標籤）。點擊盾牌圖示按鈕開啟設定 Modal，可設定：
  • 廠商驗證開關：開啟後，此群組的白名單申請在執行前會先驗證廠商是否在允許清單內，不符合則靜默拒絕。
  • 允許廠商前綴（逗號分隔，不分大小寫）：填入廠商名稱或前綴。比對採分隔符號感知模式——前綴 P 僅允許廠商名稱完全等於 P、或以 P_（底線）、P-（破折號）開頭；不會誤放行恰好含同樣字母但屬不同廠商的名稱（例如填 ON9 只允許 ON9、ON9_sub，不會放行 On9gaming）。填 TitanTR1 則允許 TitanTR1、TitanTR1_SCFLY 等旗下子品牌。
未填允許前綴或關閉開關時，此群組不受廠商限制，任何廠商都可加白。
廠商驗證開關預設關閉 新加入的群組預設不啟用廠商驗證，若需要限制特定群組只能提交特定廠商的白名單申請，需手動到此頁面設定，避免遺漏。

AI機器人救援 Tab

逐機器人設定「AI 救援」功能的啟用與逾時時間，詳細運作機制見 AI機器人救援（運作細節）。

按鍵 / 欄位功能說明 / 限制
啟用救援 開關開啟後：若群組 AI 問答已關閉，且該群組最後一則訊息逾時仍未被人工「引用回覆」，AI 會自動判斷是否為問題並接手回覆一次。
逾時時間（分鐘）可調整範圍 1 ～ 1440 分鐘；只有啟用救援時此欄位才可編輯。
狀態標籤顯示「未啟用」或「監控中（N 分）」。

Notify管理 Tab

設定「Freshdesk 工單建立通知」的接收機器人與目標群組。每當機器人在一般模式下自動回覆（關鍵字規則、知識庫問答、白名單成功處理），背景執行緒會嘗試建立 Freshdesk 工單；不論建立成功或失敗，都可透過此設定讓指定機器人自動把結果通知到指定的 Telegram 群組。

按鍵 / 欄位功能說明 / 限制
通知開關關閉時完全不發送任何工單通知；開啟後，每筆 Freshdesk 建單動作（成功或失敗）都會觸發一則通知訊息。
發送通知的機器人選擇用哪一個 Telegram 機器人來傳送通知訊息；需先在「機器人管理 Tab」建立並啟用機器人後才會出現於此下拉選單。
通知目標群組選擇要把通知發到哪個群組；下拉選單來源為所選機器人曾有訊息活動的群組清單（若目標群組不在列，需先讓機器人在該群組收到至少一則訊息）。選定後會顯示群組 ID 供確認。
儲存設定儲存後立即生效，下次工單建立時即會發送通知到新設定的目標。
全域單一設定 Notify 設定是整個系統共用一組，不區分機器人——所有機器人的 Freshdesk 工單通知都會發到同一個目標群組。若需要按機器人分開通知，目前需手動更換設定。

通知訊息格式

工單建立成功時：
群組名稱：（觸發回覆的 Telegram 群組名稱）
建立狀態：✅ 成功
工單編號：#12345
工單建立失敗時：
群組名稱：（觸發回覆的 Telegram 群組名稱）
建立狀態：❌ 失敗
失敗原因：HTTP 500: Internal Server Error
通知不觸發的情境 管控模式下由人工手動發送的回覆、Fallback 無解答回覆、以及 AI 救援回覆，都不會建立 Freshdesk 工單，因此也不會觸發 Notify 通知，詳見 訊息處理總流程。

訊息處理總流程（機器人判斷機制）⭐

這是整個系統最核心的邏輯：機器人每收到一則文字訊息，會依序通過以下判斷關卡，只要中途符合任一條件就會直接結束（return），不會繼續往下判斷。以下按實際程式碼執行順序說明。

0
忽略名單檢查
比對發送者的 user_id 或 @username（不分大小寫）是否命中該機器人「忽略名單」中已啟用的項目。
命中 直接結束，完全不回覆、不留任何統計或對話紀錄
未命中 繼續下一步
0.5
後台白名單自動處理（僅該機器人開啟此功能時執行）
偵測訊息是否為「後台 IP 加白」請求（關鍵字＋IP 格式皆符合），符合則自動登入內部白名單管理系統執行加白，詳見 後台白名單處理頁面。
至少一組帳號成功 回覆固定文字 Done，並背景建立 Freshdesk 工單
廠商驗證不通過 解析出的廠商名稱不在此群組的允許清單內 → 不執行加白、不建工單；依訊息語言回覆固定訊息（中文：「您好，人員將會協助確認，請稍後」／英文：「Hello, our team will assist you shortly. Please wait.」），詳見 群組管理 Tab
全部失敗 / 無法解析 靜默不回覆、不建工單

⚠️ 此步驟不受「即時對話管控模式」影響——即使該機器人開啟了管控模式，只要訊息符合白名單請求格式，機器人仍會自動處理並直接回覆，不會進入待審佇列。

1
關鍵字規則比對
依序檢查該機器人所有「已啟用」的關鍵字規則，只要訊息文字（不分大小寫）包含某條規則的關鍵字即算命中（取第一條符合的規則，不會同時觸發多條）。

雙語回覆判斷：命中後，系統會檢查提問是否含有中文字元（CJK 字符範圍）：
• 提問含中文 → 使用「中文回覆」
• 提問純英文 + 該規則有填「英文回覆」 → 使用英文版本
• 提問純英文 + 未填英文回覆 → 仍使用中文回覆（不會空白）

管控模式開啟 記錄訊息 + 建立「待發送回覆」（已套用語言判斷），等待人工於即時對話頁面確認發送
一般模式 直接回覆（對應語言版本）＋記錄群組統計＋背景建立 Freshdesk 工單
皆未命中 繼續下一步
2
短訊息過濾 ＋ 申請表單格式偵測
(a) 字數限制：去除頭尾空白後訊息長度 小於 10 字元，且未命中關鍵字規則者，直接略過（不回覆、不記錄），避免機器人對「嗨」「OK」「謝謝」等短句反應。
(b) 申請表單偵測：若訊息中有 3 行以上符合「數字編號＋句點/頓號＋內容＋冒號＋內容」格式（例：1. 商戶名字：ABC），視為使用者在提交結構化申請資料，而非提出問題。
字數不足 10 直接結束，無任何回應
判定為申請表單 跳過知識庫查詢，直接回覆制式訊息（依內容含中文與否自動切換中／英文），並記錄為「無解答對話紀錄」
3
該群組 AI 問答開關檢查
查詢此聊天室（chat_id）在「群組管理」中的 ai_enabled 設定（預設開啟）。
關閉 不查詢知識庫、不回覆；若此機器人有啟用「AI 救援」，會記錄此訊息為救援候選（除非本訊息本身是「引用回覆」某則候選訊息，此時代表已被人工處理，改為標記該候選為已處理）
開啟 繼續下一步
4
知識庫 AI 查詢
將該機器人所有「已啟用」知識庫文件的全部段落，連同問題一併送給 Claude Haiku，由 AI 直接判斷是否有「直接對應」的答案（不做向量相似度篩選，而是把候選片段全部交給 AI 自行判斷語意相關性）。

系統提示詞明確要求 AI：① 只回答知識庫裡有直接對應答案的內容，不可推論／整合／延伸；② 若訊息本身是表單提交格式，一律視為找不到答案；③ 必須用「與提問相同的語言」回答；④ 找不到答案時僅需簡短告知，不需要多做解釋。

AI 給出答案 進入下一步分流
AI 回覆「找不到相關資訊」等語句 視為無答案，進入 Fallback
5
依「即時對話管控」總開關分流
此開關（每個機器人各自獨立）決定知識庫查詢後的處理方式：
管控模式開啟 不論 AI 有沒有找到答案，都先把使用者訊息記錄下來；若 AI 有答案，同時建立「待發送回覆」供人工審核；管理員需至即時對話管控頁面手動按「發送」才會真正送出
一般模式（預設） AI 有答案 → 立刻自動回覆＋記錄統計＋寫入對話日誌＋背景建立 Freshdesk 工單；AI 無答案 → 進入 Fallback 流程
6
Fallback（無解答）與冷卻機制
僅在「一般模式」且知識庫無答案時觸發。以「機器人＋發送者（或聊天室）」為單位設有 6 秒冷卻：同一人在 6 秒內第二次觸發 Fallback 會被直接忽略（不回覆、不記錄），避免使用者連續發訊息時被機器人洗版式回覆。
冷卻中 直接結束，無任何回應
未冷卻 回覆制式等待訊息（中文訊息回中文、否則回英文），記錄群組統計，並寫入「AI 無解答對話紀錄」供之後補進知識庫
關於「Freshdesk 工單自動建立」 只有在一般模式（非管控模式）下，機器人「自動送出」的關鍵字回覆、知識庫回覆、白名單成功處理這三種情境，才會在背景執行緒呼叫外部 Freshdesk 建單 API 留存記錄；管控模式下由人工發送、以及 Fallback 無解答回覆、AI 救援回覆，都不會建立工單。

關鍵字規則頁面

側邊選單：Telegram 機器人 → 關鍵字規則

設定「訊息中只要包含指定文字，就直接回覆固定訊息」的規則，優先權高於知識庫 AI 問答（見 流程步驟 1）。

按鍵 / 欄位功能說明
選擇機器人規則綁定特定機器人，須先選擇才能查看／新增該機器人的規則。
新增規則填寫「關鍵字」、「中文回覆訊息（預設）」，以及選填的「英文回覆訊息」。比對邏輯為「文字包含」（不分大小寫），不是完全比對，也不支援正規表示式。
中文回覆（預設）必填。中文提問、或未填英文版時，一律使用此欄位的內容回覆。
英文回覆（選填）選填。僅當提問內容完全不含中文字元時才使用；未填此欄則所有問題皆回覆中文版本。適合填入與中文版本對應的英文說明，確保英文使用者能看懂回覆內容。
啟用開關暫時停用某條規則但不刪除。
編輯 / 刪除刪除有二次確認彈窗。

語言判斷邏輯（與 Fallback 相同規則）

提問內容使用回覆
含中文字元（例：「我想問優惠」）中文回覆
中英混合但含中文字元（例：「whitelist IP 申請」）中文回覆（含任何中文字元即視為中文）
純英文（例：「Please check my whitelist」）且有填英文版英文回覆
純英文但未填英文版中文回覆（fallback，不會空白）
不消耗 AI Token 關鍵字規則命中時，語言判斷僅靠正規表示式偵測是否含中文字元，完全不呼叫 Claude API，因此不計費也不增加延遲，秒回且零成本。
規則比對順序 目前是依資料庫回傳順序逐一比對，命中第一條就停止；同一機器人若有多條關鍵字彼此包含（例如「優惠」與「優惠活動」），實際觸發哪一條會依清單順序而定，建議關鍵字設計上盡量避免重疊。

知識庫管理頁面

側邊選單：Telegram 機器人 → 知識庫管理

分為三個分頁：知識庫來源、AI對話日誌、AI無解答對話紀錄。

知識庫來源 Tab

按鍵 / 欄位功能說明 / 限制
上傳文件選擇綁定機器人 + 檔案，允許格式：.pdf .doc .docx .xls .xlsx .txt .csv .md。上傳後立即解析並分段存入資料庫。
手動新增 Q&A不透過檔案上傳，直接在畫面上新增一組「問題／其他說法關鍵字／回答」，即時生效。
文件啟用開關關閉後，該文件所有段落不再參與知識庫查詢，但資料仍保留。
下載下載原始上傳檔案。
換機器人將整份文件重新綁定到另一個機器人。
刪除刪除文件本體與其所有段落 / Q&A。
Q&A 搜尋框在已選取文件的 Q&A 清單內，即時篩選問題／關鍵字／答案文字。

文件解析規則

  • Excel（.xls/.xlsx）：固定以 A 欄＝問題、B 欄＝答案，逐列自動轉為 Q&A。
  • PDF / Word / TXT 等：依「Q&A 文字格式」解析，格式為
    Q1: 問題標題 → （可選）其他說法或關鍵字 → A1: 回答內容，每組之間以一整行 10 個以上的破折號 ---------- 分隔。
  • 若文件不是 Q&A 格式，則改用「段落切塊」方式：中文內容每 400 字一段、重疊 50 字；英文內容每 400 字一段、重疊 40 字，讓 AI 查詢時仍有上下文可用（但不會被記成 Q&A 筆數）。
  • 查詢知識庫時，該機器人所有啟用文件的全部段落都會一次送給 Claude 判斷，沒有做關鍵字先篩選再送 AI 的兩段式流程，因此文件量越大、Token 成本越高，建議定期清理不需要的舊文件。

此功能用於自動化處理群組內常見的「後台 IP 加白名單」請求，省去人工登入內部系統手動新增的步驟。

最近 10 筆處理記錄顯示群組名稱、比對到的廠商名稱、IP 列表、成功/失敗狀態與時間，方便追查自動化是否正確判斷廠商。

自動偵測與比對規則

  1. 觸發關鍵字：訊息需包含如 whitelist bo ip、bo ip、backend ip、加白后台ip、後台ip、backoffice ip 等後台相關字樣，且訊息中同時含有至少一個 IPv4 位址才會被視為請求。
  2. 排除規則：若訊息中出現 api ip、api whitelist、加白api 等「API IP」相關字樣，一律視為非本功能處理範圍，即使同時滿足第 1 點也不會觸發。
  3. 帳號與廠商解析：從訊息中擷取 Username / 帳號 / 后台账号 / ID 等標籤後方的帳號字串，並以底線／破折號拆解各段；一則訊息可能包含多組帳號（多行列出），會逐一嘗試比對廠商。
  4. 廠商比對（四層漸進式演算法）：登入內部白名單管理後台取得完整廠商清單後，依序嘗試下列四個步驟，一旦唯一確認廠商即停止：
    • Step 4a 前綴逐層比對：從帳號第一段開始，每次多加一段組成前綴，篩選廠商名稱等於或以該前綴為起頭的清單。若某層剛好只剩一筆即命中；若某層結果為 0 筆，嘗試用上一層候選清單再做 fallback（回推帳號完整名是否以候選廠商名稱為起頭），唯一則命中。
    • Step 4b 第一段前綴反向比對：若 4a 仍未確定，改以帳號第一段為基準，尋找廠商名稱是該第一段前綴的所有廠商（即廠商名 ⊆ 帳號第一段），取名稱最長且唯一者。
    • Step 4c 段落成員比對：若 4b 仍未確定，將所有廠商名稱以底線／破折號拆成多段，找出其中任一段等於帳號第一段的廠商。例如帳號 SCFLY_spinpokies29_MYR 第一段 SCFLY 是廠商 TitanTR1_SCFLY 的第二段，此步驟即可命中；取最長且唯一者。
    • Step 4d 群組廠商驗證：確認廠商後，若此群組已在「群組管理 Tab」開啟廠商驗證並填有允許前綴，則判斷解析到的廠商名稱是否以允許前綴之一開頭；不符合則視為本群組無權提交該廠商的加白申請，靜默拒絕（詳見下方「群組廠商驗證」說明）。
解析成功 ≥1 組帳號 呼叫內部系統新增白名單 IP，機器人回覆 Done，並建立 Freshdesk 工單留存紀錄
廠商驗證不通過（Step 4d） 解析到的廠商不屬於此群組的允許廠商 → 不加白、不建工單；依訊息語言自動回覆固定訊息（中文：「您好，人員將會協助確認，請稍後」／英文：「Hello, our team will assist you shortly. Please wait.」）
全部失敗（廠商無法判斷 / 登入失敗等） 機器人不回覆任何訊息，避免誤導使用者以為已處理完成；僅在記錄檔中留下失敗 log

群組廠商驗證（Step 4d）

為防止不同群組的代理將加白請求送到錯誤群組（例如 B 廠商代理在 A 廠商群組發送申請），每個群組可獨立設定「允許廠商前綴」白名單：

欄位說明
廠商驗證開關 預設關閉（不限制）。開啟後，系統在完成廠商比對（Steps 4a–4c）後會執行前綴驗證，不符合則靜默拒絕整筆請求。
允許廠商前綴（逗號分隔） 填入允許的廠商名稱或前綴，不分大小寫。採分隔符號感知比對，前綴 P 通過條件為以下三者之一：
• 廠商名稱 完全等於 P
• 廠商名稱以 P_（底線）開頭
• 廠商名稱以 P-（破折號）開頭

範例：
• 填 ON9 → 允許 ON9、ON9_subagent，但不放行 On9gaming（屬於不同廠商，名稱中間無分隔符號）
• 填 TitanTR1 → 允許 TitanTR1、TitanTR1_SCFLY 等旗下子品牌
• 留空（或關閉開關） → 不做任何限制，任何廠商皆可加白
設定入口：群組管理 Tab → 找到目標群組 → 點擊「🛡」按鈕 → 開啟「廠商驗證」開關並填入允許前綴後儲存。每個群組可設定不同的允許清單，互不影響。
帳密與內部系統資訊 自動化所使用的內部系統網址、登入帳密皆透過環境變數設定（部署時於 Render / 本機 .env 配置），不會寫在程式碼明碼或此文件中，請勿將相關金鑰外流。

AI機器人救援（運作細節）

設定入口見 機器人管理頁面 → AI機器人救援 Tab，這裡說明背景真正如何運作。

設計目的 當某個群組把「AI 問答」關掉、改由真人客服負責時，若真人一時忙碌沒有即時回覆，AI 救援可以在「等待一段時間後仍無人回應」的情況下自動接手，避免使用者被晾在一邊太久。

背景排程

系統啟動時會啟動一個永久迴圈，每 60 秒掃描一次所有「已啟用救援」的機器人。

1
收集候選訊息
只有當某群組的「AI 問答」關閉時，該群組收到的非引用回覆訊息才會被記錄為「救援候選」；若使用者訊息是直接引用（reply）某則候選訊息，代表已被真人處理，該候選會被標記為「已處理」，不會再被救援。
2
逾時篩選
每輪掃描找出「建立時間已超過設定的逾時分鐘數、且尚未處理」的候選訊息；同一群組若有多則，只取最新一則處理，其餘舊候選一併標記為已處理（不會逐則都救援）。
3
AI 判斷是否為「問題」
呼叫 Claude Haiku 只回答 yes/no，排除單純問候、感謝、簡短回應（例如「謝謝」「你好」「OK」「收到」）等不需要回答的訊息，避免對這類訊息也強行回覆。
4
查詢知識庫並回覆
與一般流程相同的知識庫查詢邏輯；有答案就回答案，沒有答案則發送制式等待訊息（依語言自動切換）。回覆時會引用原始訊息，讓使用者清楚知道是回覆哪一則。
限制條件
掃描頻率每 60 秒一輪，因此實際觸發時間點與設定的「逾時分鐘」相比會有最多 1 分鐘誤差
逾時可調範圍1 ～ 1440 分鐘（1 分鐘 ～ 24 小時）
前提條件該機器人啟用中 且 該機器人啟用救援 且 該群組 AI 問答已關閉
每群組每輪只救援最新一則候選，避免一次回覆一長串堆積的舊訊息
不受管控模式影響AI 救援的回覆是直接呼叫 Telegram API 送出，不會進入「待發送回覆」審核佇列
